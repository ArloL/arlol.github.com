1 Introduction

The average number of electronic devices used in everyday and professional life is increasing. With this increasing number of heterogenous devices the application model of a user sitting in front of one device to achieve a task is outdated. Even less so for team situations. A user should be able to make use of each device’s capabilities to increase his productivity. While data manipulation is feasible on a laptop, detailed consumption is most efficient on a tablet, the display of large data makes most sense on a TV and team collaboration works best on a touch table. To make use of these capabilities, applications need to be able to interact with each other across devices. As current available approaches are either proprietary, focused on a single task or too difficult (see chapter 1.2.3) this thesis presents a general approach that allows developers to easily create interacting applications using standardized web technologies and the REST architecture style.

Even though there are many possible applications (see chapter 1.2.2) research is currently not investigating the possibility of creating an open general solution for interacting applications as seen in chapter 2. To reduce the necessary design and development time of interacting applications this thesis will contribute an open source framework enabling Device Interaction [DI] called IntAirAct.

It is crucial that the framework uses open standards. This not only reduces the design and implementation time of the framework but also ensures cross-platform compatibility.

Due to the ubiquity of the Internet there are many frameworks and implementations for all major platforms available that use standardized web technologies (e.g. HTTP or JSON). Especially for communication on mobile devices web technologies are widely used. This makes them an ideal candidate for the communication layer. As will be seen (see chapter chapter) the architecture style REST is also feasible for DI.

To overview the structure of this thesis: First the requirements of the framework will be investigated followed by the design of it. Secondly it will be implemented for two major platforms: iOS/OS X and C#. Finally the framework will be evaluated.

This chapter will continue to cover the background necessary to understand the contributions of this thesis. First DI will be formally defined followed by the major challenges when trying to create interacting applications. Research questions and goals are also stated.

1.1 Device Interaction

First it is necessary to define what this thesis understands as DI. The definition is then supported and the usefulness of DI motivated by presenting three example applications.

1.1.1 Definition

Interaction is defined as “a mutual or reciprocal action or influence”.1 This definition can be extended to serve the purposes of this thesis:

Device Interaction is a mutual or reciprocal action or influence between devices.

To understand the entirety of the definition the parts of it will be defined in more detail.

First of all it is necessary to understand the term device. A device is any electronic device. This includes - but is not restricted to - smartphones, tablets, computers and touch tables.

Next it is important to understand what is happening between these devices. An action or influence is the ability of a device to access and/or change the state of another device.

The notion of mutual or reciprocal means that the actions can be exchanged bi-directionally between the devices.

In sum this means that all devices have the ability to access and/or change the state of all the other devices. Thus allowing applications to cross the boundaries of a single-device.

In order to concretize the definition the next chapter presents three example applications.

1.1.2 Example applications

This chapter will present two existing proprietary applications that fall under the definition of Device Interaction. As far as known and relevant implementation details will be explained. Finally the project for which IntAirAct was created will be presented.

1.1.2.1 Remote for iOS

Remote for iOS is an application that allows the remote control of a media player2 running on a desktop computer using a mobile device3.

A screenshot of Remote for iPad

This example demonstrates the bi-directional nature of device interaction: while the mobile device displays information from the desktop computer (e.g. the available songs) the mobile device controls the desktop computer (e.g. play this song). The protocol used4 is proprietary but has been partly reverse engineered. The system uses the HTTP protocol for requests and a binary protocol for the responses.

1.1.2.2 Bump

Bump is an application that allows the exchange of information between devices using a shake gesture.

Bump being used to exchange contact information

This example demonstrates the technological issues of interacting applications. Bump uses an Internet connection to transfer data between two devices that are in close vicinity. This is very inefficient as the data has to move from the devices to a server and back.

1.3 Research questions

This thesis presents a new approach for device communication that allows interacting applications.

Given the fact that the architecture is not implemented for all major platforms: 1. Is it possible to be implemented on all major platforms?

Since bi-directional communication requires a web server running on all devices on of the research questions is: 1. Is it possible to have web servers on mobile devices?

Since web technologies have been created for long-distance high-latency communication they include overhead that is not necessary in local-area scenarios. Thus the performance of the framework has to be evaluated: 2. Is the performance sufficient?

1.4 Goals

There are two main goals of this thesis. The first is to design an architecture for Device Interaction [DI]. The second is to implement the stack in form of a framework. The third goal is to attempt to answer the research questions listed above to show that the approach followed while developing the framework is feasible for creating interacting applications.

3 Requirements

I think it makes sense to distinguish between architecture and framework. The general architecture has different requirements than the actual implementations.

Architecture

Functional

This is essentially the same as the requirements of a MOM. Look at them from Schulze.

Communication

The communication is a fundamental part of the architecture. It has to allow for peer-to-peer communication to allow direct interactions of the devices without a middleman. The communication has to be bi-directional to allow true exchange between the devices.

Device discovery

To enable Ad-Hoc interaction scenarios it is essential that the devices can discover each other without requiring infrastructure in the sense of an internet connection or a setup server. The devices should be able to discover each other without a middleman. This reduces set-up cost and allows for use in many everyday situations like a coffeeshop.

Addressing Scheme

It is necessary to uniquely identify devices in the network of devices to allow for direct communication.

Resource

The basis of the architecture is to be able to share resources (images, documents) between devices. The architecture should define how these resources are identified.

Cross-platform compatibility

One of the goals is to support all modern devices. Here is a list of the devices that are meant by “modern device”. Additionally all the Operating Systems are listed as well as the programming languages that have to be supported by the framework. This means that the technologies used have to be compatible with the majority of these platforms.

  • Mobile devices
    • Tablets
      • iPad (iOS >= 4.0)
      • Android Tablets (Android >= 2.3)
    • Phones
      • iPhone (iOS >= 4.0)
      • Android Phones (Android >= 2.3)
      • Windows Phone (>= 7)
      • Blackberry
  • Touch Tables
    • Microsoft Surface
  • TV
    • Restriction: with a computer/media center connected
    • Apple TV (uses OS X, but is not an open platform)
    • Boxee
    • XBMC

  • Sound station
    • Restriction: with a computer/media center connected

Performance

To allow natural interactions between the devices the architecture has to deliver sufficient performance to not distract the user. The 100ms boundary introduced by Robert B. Miller5 should be sufficient. To allow for situations like congestion, etc. the goal of the architecture is 10ms / message.

Flexibility

If necessary the application might decide to utilise a different communications protocol like RTSP+RTP.

Usability

The architecture as well as the framework should stay out of the way of the user as much as possible. The implementations should require as little code as possible.
